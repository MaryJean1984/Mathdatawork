# 🏆 核心项目亮点：企业级数据库高阶架构与精进设计

为了使本项目在评审中脱颖而出，拿到“满分/极优”评价，我们在基础的 CRUD 与三层缓存架构之上，从**数据库底层视角**进行了深度的企业级架构精进。以下五大亮点可直接作为项目答辩或简历中的核心技术卖点。

## 亮点一：基于 MySQL 8.0 原生 JSON 的 AI 非结构化数据存储
**业务痛点**：传统的单词表需要固定的字段（如 `collocation1`, `collocation2`），但接入牛津字典和 AI 大模型后，返回的“词语搭配”、“同义词”、“多场景例句”等数据长度和结构极不固定，传统关系型字段难以兼容。
**架构精进**：
- 深度应用 MySQL 8.0 的原生 `JSON` 数据类型，在 `word_library`（词库表）和 `ai_word_record`（AI交互表）中新增 JSON 字段存储富文本结构。
- **技术优势**：既保留了关系型数据库的事务一致性，又获得了类似 MongoDB 的文档型扩展能力。通过 MySQL 的 `JSON_EXTRACT()` 和 `->>` 语法，可直接在 SQL 层面高效解析 AI 生成的多维数组数据。

## 亮点二：冷热数据分离与 MySQL 表分区 (Table Partitioning)
**业务痛点**：随着用户每日打卡背诵（包含海量的高频错题和听写记录），`study_record`（学习记录表）的数据量会呈指数级暴增，导致历史数据拖慢当天的查询速度。
**架构精进**：
- 引入**冷热数据分离架构**，利用 MySQL 的 `PARTITION BY RANGE` 核心特性，对学习记录表进行按“时间维度（月/季度）”的物理分区。
- **技术优势**：将 3 个月前的历史背诵数据自动划分到“冷分区”，当月的数据留在“热分区”。查询当月统计数据时，MySQL 优化器会自动触发 **Partition Pruning (分区裁剪)**，直接跳过海量历史数据，极大提升 I/O 性能。

## 亮点三：终极索引优化：覆盖索引与索引下推 (ICP)
**业务痛点**：普通项目仅停留在“加单列索引”的层面，在复杂的多条件筛选（如：查询某语种下、某学习阶段的特定用户）时，依然会产生大量回表（回表查询聚集索引），引发性能瓶颈。
**架构精进**：
- 落地 **覆盖索引 (Covering Index)**：在 `sys_user` 表建立联合索引 `idx_lang_stage_name (language, stage, username)`。当系统仅需查询用户名列表时，所需数据直接在非聚集索引树上获取，实现 `Using index`（零回表）。
- 落地 **索引下推 (Index Condition Pushdown, ICP)**：在单词多条件模糊检索时，将原本在 Server 层进行的过滤条件下推至 InnoDB 存储引擎层执行，大幅减少回表次数，执行计划中清晰体现 `Using index condition`。

## 亮点四：千万级深分页终极优化：延迟关联 (Deferred Join)
**业务痛点**：在管理员后台管理数万用户信息，或者浏览几万词汇库时，`LIMIT 100000, 20` 这种深分页会导致 MySQL 扫描 100020 行数据后丢弃 10 万行，产生致命的性能灾难。
**架构精进**：
- 摒弃传统写法，采用 **延迟关联 (Deferred Join)** 策略重写 SQL。
- **原 SQL**: `SELECT * FROM word_library ORDER BY id LIMIT 100000, 20;`
- **优化后 SQL**: `SELECT w.* FROM word_library w INNER JOIN (SELECT id FROM word_library ORDER BY id LIMIT 100000, 20) as tmp ON w.id = tmp.id;`
- **技术优势**：子查询只查询主键 `id`，完美利用覆盖索引（无回表），快速定位到第 10 万行的主键，然后再与主表进行 20 次精确的 INNER JOIN，将查询耗时从秒级降低至毫秒级。

## 亮点五：三层缓存架构下的“缓存一致性”保障方案
**业务痛点**：系统引入了 Redis 和 C++ 手写 MemoryDB，当用户修改个人资料或单词词义更新时，如何保证 MySQL 数据库与两层缓存中的数据一致，是高阶系统必须解决的难题。
**架构精进**：
- 引入 **延迟双删 (Delayed Double-Delete) 策略**：
  1. 更新数据前，先删除 Redis / MemoryDB 缓存。
  2. 更新 MySQL 数据库。
  3. 异步线程/协程休眠 500ms（等待数据库主从同步或事务提交）。
  4. 再次发起删除缓存操作。
- **技术优势**：彻底解决了高并发场景下，由于读写线程交替执行导致的“缓存脏数据”问题，体现了极其严谨的企业级并发架构思维。
